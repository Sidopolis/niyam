## What is this?

Niyam is a CLI and MCP server for AI coding agents. It gives you 50 modular rule files that you compose into a single focused context for your tool (Kiro, Cursor, Claude Code, Copilot, Windsurf, Codex, Aider, Gemini).

The MCP server watches which file you're editing and serves only the rules that apply to that file. No static 500-line dumps.

---

## Install

```bash
npx niyam init
```

This scans your project, asks what roles and principles you care about, and writes a config. The output goes to `AGENTS.md`, `.cursorrules`, or `CLAUDE.md` depending on your tool.

```bash
# Add modules to an existing config
npx niyam add react typescript tdd

# Pull the latest version of your subscribed rules
npx niyam update

# See what the composed output looks like
npx niyam preview

# Regenerate after editing config
npx niyam generate --tool cursor

# Search all available rules
npx niyam search "security"

# List everything
npx niyam list
```

## MCP Server

The MCP server resolves rules dynamically based on the file your agent is working on.

```
src/components/Button.tsx  →  react, typescript, frontend, accessibility, clean-code
src/api/auth.ts            →  typescript, backend, security-first, clean-code
Dockerfile                 →  docker, devops
prisma/schema.prisma       →  prisma, backend
tests/user.test.ts         →  typescript, testing, tdd, clean-code
```

It also stores project-specific corrections. If you tell your agent "we use pnpm here," it remembers that for future sessions.

### Setup

Add to your MCP config (Kiro: `~/.kiro/settings/mcp.json`, Cursor: `.cursor/mcp.json`):

```json
{
  "mcpServers": {
    "niyam": {
      "command": "node",
      "args": ["path/to/niyam/mcp/src/index.js"]
    }
  }
}
```

### Tools

| Tool | What it does |
|------|-------------|
| `get_rules` | Returns rules for the file being edited |
| `get_checklist` | Returns a verification checklist for the current context |
| `get_context` | Full bundle: rules + learnings + checklist |
| `add_learning` | Saves a correction for future sessions |
| `get_learnings` | Shows all stored corrections |
| `remove_learning` | Deletes an outdated correction |

## Rules

50 modules across 6 categories. Each one is 100-200 lines of specific, version-targeted guidance.

| Category | Count | Examples |
|----------|-------|---------|
| **stacks/** | 23 | react, nextjs, python, go, rust, flutter, vue, angular, docker, aws, prisma, supabase, drizzle, astro |
| **roles/** | 12 | frontend, backend, devops, security, architect, ai-engineer, qa-tester, technical-writer |
| **principles/** | 6 | clean-code, tdd, ddd, security-first, performance, accessibility |
| **workflows/** | 5 | git, ci-cd, testing, deployment, code-review |
| **anti-patterns/** | 2 | common, ai-slop |
| **languages/** | 2 | javascript, sql |

### Templates

If you don't want to pick individually:

```bash
npx niyam init --template startup-saas
npx niyam init --template enterprise
npx niyam init --template solo-dev
npx niyam init --template open-source
```

## How the output is structured

```markdown
<!-- Generated by Niyam v0.1.0 -->
<!-- Modules: react, typescript, frontend, clean-code, git -->

# Project Rules (Niyam)

## Active Modules
- Stacks: react, typescript
- Roles: frontend
- Principles: clean-code
- Workflows: git

---

## Principles
[clean-code rules]

## Tech Stack
[react rules]
[typescript rules]

## Role Guidelines
[frontend rules]

## Workflows
[git rules]
```

Principles come first (universal), then stack-specific, then role-specific, then workflows. Only what you subscribed to shows up.

## Project structure

```
├── stacks/          23 tech-specific rule modules
├── roles/           12 role-based rule modules
├── principles/       6 engineering principle modules
├── workflows/        5 workflow modules
├── anti-patterns/    2 what-not-to-do modules
├── languages/        2 language-specific modules
├── templates/        4 pre-configured combos
├── examples/         3 example configs with generated output
├── cli/             CLI source (zero external deps)
├── mcp/             MCP server (@modelcontextprotocol/sdk)
├── tests/           22 tests (node --test)
└── web/             Website source (deployed separately)
```
